# Cornerstone OS — Live Demo Access Flow

This document summarizes the **industry selection, email capture, and temporary guest session** flow that lets visitors explore seeded demo environments without creating an account or using the normal login page.

---

## 1. Full demo entry flow

1. **Try the live demo**  
   Visitor clicks “Try the live demo” (or equivalent CTA) on the marketing site.

2. **Industry selection modal**  
   A modal opens: “Choose your industry to explore the Cornerstone demo” with four cards:
   - Facility Maintenance Companies  
   - Industrial / Manufacturing Maintenance  
   - School District Maintenance Teams  
   - Healthcare Facility Maintenance Teams  
   Desktop: 2×2 grid. Mobile: stacked cards. “Just show me the platform” links to `/how-it-works` and does not start the demo flow.

3. **Email capture modal**  
   After selecting an industry, the modal shows: “Launch your live demo” with:
   - Work email (required)  
   - Company name (optional)  
   - “Enter Demo” button  
   Helper text: “No scheduling required. Explore a live environment with realistic seeded data.”

4. **Temporary guest session**  
   On submit, the server:
   - Creates or reuses a Supabase Auth user for that email (no password shown).
   - Adds a `tenant_membership` with role `demo_guest` for the chosen industry’s demo tenant.
   - Inserts a row into `demo_leads` (email, company_name, industry_slug, timestamp).
   - Generates a magic link for that user and returns its URL.

5. **Redirect into the app**  
   The client redirects the user to the magic link. Supabase verifies the link and sends the user to `/auth/callback?next=/dashboard`, where the session is established and they are redirected to `/dashboard`. They never see the login page.

6. **Welcome modal (once per session)**  
   Inside the app, if the user is a demo guest and has not seen the welcome modal this session, a modal is shown: “Welcome to the Cornerstone demo” with:
   - “Start guided walkthrough” → can start a tour.
   - “Explore on my own” → dismiss.  
   The choice is stored in `sessionStorage` so the modal only appears once per session.

**Direct demo URLs**  
Visitors can also open a demo by industry directly, e.g.:
- `/demo/facility-maintenance`
- `/demo/industrial`
- `/demo/school-district`
- `/demo/healthcare`  

These pages show the same email capture form (industry fixed). Submitting the form runs the same server logic and redirect flow as above.

## 2. Industry routing logic

- **Config**  
  Industry slug → tenant and label is defined in `lib/marketing-site.ts` in `DEMO_LOGIN_CONFIG` and `INDUSTRY_DEMO_OPTIONS`:
  - `facility-maintenance` → tenant slug `summit-facility-demo`
  - `industrial` → `northstar-manufacturing-demo`
  - `school-district` → `riverside-schools-demo`
  - `healthcare` → `mercy-healthcare-demo`

- **Flow**  
  The selected (or URL) industry slug is sent with the email form. The server action `enterDemoAction` in `app/demo/actions.ts` looks up the tenant by slug, creates/updates the guest user and membership, stores the lead, and returns the magic link for redirect.

- **Seeded data**  
  Demo tenants and data are created by `npm run seed:demo`. No new demo data is generated by the demo access flow itself.

---

## 3. How temporary guest sessions are created

- **Auth user**  
  Server-side (admin client) creates a Supabase Auth user with the submitted email and a random password (never sent to the client). `email_confirm: true` so the user is immediately valid.

- **Tenant membership**  
  A row is inserted (or upserted) in `tenant_memberships` with `role = 'demo_guest'` for the demo tenant. The existing RLS and app logic use this membership to scope data to that tenant.

- **Sign-in**  
  The server calls `auth.admin.generateLink({ type: 'magiclink', email, options: { redirectTo } })` and returns the `action_link` to the client. The client redirects the user to that link; Supabase verifies and redirects to `/auth/callback?next=/dashboard`, where the session is established via `exchangeCodeForSession` (or existing session from hash). No shared demo password is exposed; each visitor gets a one-time magic link.

- **Persistence**  
  The guest is a normal Auth user with a single `demo_guest` membership. Sessions follow the existing Supabase/Next.js cookie-based session handling. Demo environments can be reset by reseeding (e.g. `npm run seed:demo:refresh`) and optionally cleaning or rotating demo guest users.

## 4. Where lead data is stored

- **Table**  
  `public.demo_leads` (see migration `20260321000000_demo_guest_and_leads.sql`).

- **Columns**  
  - `id` (uuid, PK)  
  - `email` (text, required)  
  - `company_name` (text, optional)  
  - `industry_slug` (text, required)  
  - `created_at` (timestamptz)

- **When**  
  One row is inserted per “Enter Demo” submission in `enterDemoAction`, after the user and membership are created.

- **Usage**  
  Use for follow-up, analytics, or CRM. No built-in UI in the app; query or export as needed.

---

## 5. Restrictions for demo users

Demo guests are identified by having at least one `tenant_membership` with `role = 'demo_guest'` (helper: `isDemoGuestUser()` in `src/lib/auth-context.ts`).

- **Settings**  
  Demo guests are redirected away from `/settings` to `/dashboard` (see `app/(authenticated)/settings/layout.tsx`). So they cannot:
  - Access billing (if/when under Settings)
  - Invite users (Users)
  - Change organization security or roles (Roles & Permissions, etc.)

- **Sidebar**  
  The “Organization” nav group (which contains Settings) is hidden for demo guests in the sidebar (`app/(authenticated)/components/sidebar.tsx`).

- **Platform admin**  
  Demo guests are not platform super admins, so they do not see platform admin or switch-tenant links.

- **Data**  
  They can view and use the app within their demo tenant (same RLS as other members). Destructive or org-level changes are limited by hiding Settings and by reseeding when needed. If further write restrictions are required, they can be added (e.g. by role in server actions or RLS) without changing the core auth flow.

## 6. Recommended future improvements

- **Auth callback and magic link**  
  Confirm Supabase’s exact redirect format (query `code` vs hash) for `generateLink` magic links and ensure `/auth/callback` handles it (e.g. `exchangeCodeForSession` or hash handling) on all environments.

- **Rate limiting**  
  Add rate limiting or abuse protection on `enterDemoAction` (e.g. by IP or email) to avoid spam or excessive guest creation.

- **Demo session expiry**  
  Optionally expire demo guest sessions or memberships after a time window (e.g. 24 hours) and/or clean up old `demo_leads` and guest users periodically.

- **Guided walkthrough**  
  Wire “Start guided walkthrough” in the welcome modal to an actual tour (e.g. existing tour system) so it starts a concrete flow.

- **Reseed strategy**  
  Document or automate a safe way to reset demo tenants and optionally remove or archive demo guest users when refreshing demo data.
